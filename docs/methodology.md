# 績效評估方法論演進紀錄

> 本文件記錄 perf-toolkit 的評估方法是如何從實際管理經驗中逐步形成的，供專案開發參考。

---

## 第一階段：框架建立

### 起因

2026 年 1 月，為了給資訊部（11人）建立正式的績效評估制度，從零開始設計框架。

### 設計過程

1. **確認評估面向** — 從部門實際工作型態出發，歸納出五大面向：

| 面向     | 權重 | 選擇理由                                       |
|----------|------|------------------------------------------------|
| 專案達成 | 30%  | 部門以專案開發為主，這是最核心的產出指標       |
| 技術成長 | 20%  | IT 部門需要持續學習，否則技術債會越積越多       |
| 日常維運 | 20%  | 系統穩定運作是基本功，不能只看新專案           |
| 紀律配合 | 15%  | 值班、出勤是團隊運作的基礎契約                 |
| 團隊協作 | 15%  | 11 人團隊，協作品質直接影響整體效率            |

2. **確認等第分配** — 配合公司制度，採用強制分配：
   - 特優 10%（1人）
   - 優 20%（2人）
   - 甲 50%（5人）
   - 乙 20%（2人）

3. **核心原則確立**：最終績效等級由主管綜合判斷，系統只提供參考依據。

## 第二階段：資料收集方法

### 資料來源一：里程碑績效報告（PPT）

每位成員每月更新一份里程碑績效 PPT，包含：

- 年度工作計畫（年初設定）
- 新增里程碑（年中追加）
- 每月進度更新（百分比 + 說明）

**格式**：每張投影片對應一個月份，列出該月完成的工作項目與進度。

**優點**：

- 有具體的工作紀錄可追溯
- 每月更新，不會年底才回憶

**缺點**：

- 各人詳細程度差異大（有人寫很細，有人只列標題）
- 無法反映隱性貢獻（如：顧問諮詢、跨組支援）
- PPT 格式不利於自動化分析

### 資料來源二：考勤數據

從 HR 系統匯出，包含：

- 出勤紀錄：總出勤天數、遲到次數/率/平均分鐘數、早退次數/率
- 請假紀錄：各假別次數與天數（特休、病假、事假、公出、公假等）

**分析重點**：

- 遲到率與平均遲到時間（區分「踩線型」vs「大遲到型」）
- 早退率（需排除班別設定問題）
- 公出天數（反映外勤/出差頻率，不應列為負面指標）
- 不以加班時數作為正向指標（避免鼓勵無效加班）

**踩過的坑**：

- 有成員早退率高但實際是班別設定問題 → 需個別確認背景
- 有成員遲到率極高但產出極佳 → 紀律與績效需分開看待，不因單一面向否定整體

### 資料來源三：主管觀察

最難量化但最重要的部分，包含：

- 技術成長的實際感受（誰在進步、誰在原地踏步）
- 團隊協作的真實狀況（誰好相處、誰難配合）
- 日常維運中的隱性貢獻（誰是大家遇到問題會去問的人）
- 工具採用與學習態度（例：誰主動申請 AI 工具）

**量化嘗試**：

- AI 工具申請狀況（Claude Code Max 等）→ 反映學習積極度
- 週六值班統計 → 反映紀律配合度
- 里程碑報告繳交完整度 → 反映規範遵守

## 第三階段：綜合評估方法

### 評估流程

```
里程碑報告 ──→ 專案達成評估（每人逐月檢視）
              ├→ 技術成長評估（分梯隊排序）
              └→ 日常維運評估（角色定位）

考勤數據 ────→ 紀律配合評估（遲到率、早退率、值班配合）

主管觀察 ────→ 團隊協作評估（溝通態度、跨組支援）
              └→ 各面向的修正與補充

綜合排序 ────→ 對應等第分配 ──→ 產出報告
```

### 實際評估發現

**技術成長的梯隊化**

在逐一檢視後，發現技術成長可以自然分為三個梯隊：

- 第一梯隊：主動引入新技術，對部門有帶動作用
- 第二梯隊：在專業領域持續深化
- 第三梯隊：以既有技術執行為主，較少接觸新技術

觀察指標之一：是否主動申請 AI 協作工具（部門僅 3 人申請 Claude Code Max）

**日常維運的角色差異**

不是每個人都做維運，需要區分：

- 核心維運人員（資深顧問、IT 基礎設施）
- 開發為主但有維運成分
- 純開發角色

資深成員的「被諮詢」價值在報告中很難量化，但對部門運作至關重要。

**紀律配合的客觀指標**

適合當指標的：

- 遲到率與頻率
- 週六值班配合度
- 里程碑報告繳交完整度

不適合當指標的：

- 加班時數（多不代表好）
- 純出席時間（只衡量「人在」沒意義）

需要個別確認的：

- 早退是否為班別設定問題
- 遲到是否有合理原因（通勤距離等）

**團隊協作的主觀判斷**

最難量化的面向，主要靠主管日常觀察：

- 跨組支援的意願
- 溝通態度與配合度
- 是否願意分享知識

### 報告產出

最終產出三種版本：

1. **主管版**（完整版）— 排名、考勤、詳細評語，供主管決策用
2. **公佈版**（簡化版）— 僅等第分組，無排名無負面備註，供部門公告
3. **個人版**（面談用）— 個人五大面向分析，用於一對一績效面談

## 第四階段：制度配套

### 已建立的配套制度

1. **週六值班制度**
   - 每次 2 人值班，9 人輪值
   - 每月排班一次，排定即確定
   - 違規處理：約談 → 納入年度考核

2. **加班管理規則**
   - 自主申報制，事後 3 個工作日內申請
   - 每月上限 46 小時，超時需書面報告
   - 不鼓勵無效加班

3. **「當我不在的時候」決策框架**
   - 三圈原則：自己決定 / 先做再報 / 要等核准
   - 緊急狀況 SOP
   - 溝通回報原則

4. **部門回饋機制**
   - 教育訓練後收集匿名意見
   - 彙整為行動項目追蹤

## 待改善項目（2026）

1. **里程碑報告格式統一** — 目前 PPT 格式不一，應統一結構方便自動化
2. **年中 Review 機制** — 目前只有年底一次評估，缺乏中期回饋
3. **同儕評估** — 團隊協作目前只靠主管觀察，可考慮加入同儕回饋
4. **歷年數據比較** — 目前無法對照去年表現，需要建立歷史資料庫
5. **自動化程度** — 考勤分析、報告產出應腳本化，減少人工作業

---

## 適用情境與限制

### 適用

- 5-15 人的 IT / 研發團隊
- 主管對團隊成員有直接觀察的小型組織
- 成員以專案開發 + 系統維運為主的技術團隊

### 不適用

- 大型組織（>50人）需要更正式的 HR 制度
- 純業務團隊（評估面向需要重新設計）
- 主管無法直接觀察團隊的遠端/分散式團隊
